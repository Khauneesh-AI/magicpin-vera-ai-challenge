# Magicpin Vera AI Challenge ‚Äî Project Plan v2

**Updated:** 2026-05-01
**Supersedes:** `Magicpin_Vera_AI_Challenge_Project_Plan_Final.docx`
**Reference architecture:** [cloudera/CAI_AMP_Synthetic_Data_Studio](https://github.com/cloudera/CAI_AMP_Synthetic_Data_Studio) ‚Äî `app/core/model_handlers.py`, `app/core/model_endpoints.py`

---

## 0. Why this version exists

v1 was structurally sound but kept everything in one `bot.py`. v2 refactors into a Cloudera-style package (`app/core`, `app/models`, `app/routes`, `app/services`) so:

1. The judge can read the code and immediately see separation of concerns.
2. The optional LLM polish layer can be added/removed without touching the deterministic core.
3. Tests target one module at a time.
4. Bugs found in v1 review (URL policy, suppression scope, anti-repetition, Hindi auto-reply, reply-handler reuse) get fixed during the refactor instead of bolted on later.

The shape mirrors Cloudera's `app/` tree, scaled down to what the Vera judge actually needs.

---

## 1. Working principle

> First make it correct, deterministic, fast, and judge-compatible. Then improve message quality trigger-by-trigger. Never block the hot path on an LLM.

## 3. Module-by-module spec

### 3.1 `bot.py` (public shim)

```python
from vera_bot.main import app
from vera_bot.core.composer import compose

__all__ = ["app", "compose"]
```

That's it. The judge runs `uvicorn bot:app` or imports `compose`. Everything real lives in the package.

### 3.2 `vera_bot/main.py`

- Build FastAPI app, set title/version.
- Mount `health`, `context`, `tick`, `reply` routers under `/v1`.
- On startup: if `LLM_ENABLED`, run `model_endpoints.probe()` once and log result; never block.
- Initialize `core.store.Store` as a module singleton injected via FastAPI dependency.

### 3.3 `vera_bot/core/config.py`

Plain module of typed env reads. Defaults that make sense without any env vars set:

| Var | Default | Purpose |
|---|---|---|
| `LLM_ENABLED` | `false` | Master switch for LLM polish |
| `LLM_PROVIDER` | `openai` | Which `model_handlers` branch to take |
| `LLM_MODEL` | `gpt-4o-mini` | Model id |
| `LLM_TIMEOUT_S` | `8` | Hard cap; well below the 30s `/v1/tick` budget |
| `MAX_BODY_CHARS` | `320` | Body length cap before trim |
| `URL_POLICY` | `digest_only` | `none` / `digest_only` / `any` ‚Äî see ¬ß6.2 |
| `ANTI_REPEAT_WINDOW` | `5` | Last N body hashes per merchant to block verbatim repeats |

### 3.4 `vera_bot/core/store.py`

In-memory only ‚Äî fine for the judge's test window since the process doesn't restart.

```python
class Store:
    contexts: dict[tuple[str, str], dict]                 # (scope, id) ‚Üí {version, payload, stored_at}
    conversations: dict[str, list[dict]]                  # conversation_id ‚Üí turns
    sent_suppressions: set[tuple[str, str]]               # (merchant_id, suppression_key)  ‚Üê FIX from v1
    recent_bodies: dict[str, collections.deque[str]]      # merchant_id ‚Üí last N body hashes
    auto_reply_hashes: dict[str, list[str]]               # conversation_id ‚Üí merchant turn body hashes
```

**Bug fixes baked in:**
- Suppression is `(merchant_id, suppression_key)` ‚Äî v1 used a global set, which would silently drop messages when a category-shared key like `"research:dentists:2026-W17"` was reused across 10 dentists.
- `recent_bodies` deque enforces anti-repetition (brief ¬ß11 penalty).
- `auto_reply_hashes` enables the brief's hint: "same message verbatim 3+ times = auto-reply".

### 3.5 `vera_bot/core/composer.py`

Single public function:

```python
def compose(
    category: dict,
    merchant: dict,
    trigger: dict,
    customer: dict | None = None,
    *,
    store: Store | None = None,
) -> ComposedMessage:
    ...
```

Routing:
1. If `customer` or `trigger.scope == "customer"` ‚Üí `trigger_handlers.customer_dispatch(...)`.
2. Else ‚Üí `trigger_handlers.merchant_dispatch(...)`.
3. Pass result through `validators.finalize(...)` (length, URL policy, CTA shape, anti-repeat).
4. If anti-repeat fires, request a variant from the same handler with a `seed` token; if still a dup, return the safest fallback for that trigger family.

The composer never calls the LLM directly. Only `validators.finalize` may, when `LLM_ENABLED` and the deterministic body fell into a known-weak fallback branch.

| Function | Triggers handled |

| `handle_compliance` | `regulation_change`, `supply_alert` |
| `handle_perf_movement` | `perf_dip`, `perf_spike`, `seasonal_perf_dip` |
| `handle_competitor` | `competitor_opened` |
| `handle_curious_ask` | `curious_ask_due` |
| `handle_seasonal` | `festival_upcoming`, `ipl_match_today`, `category_seasonal` |
| `handle_review_milestone` | `review_theme_emerged`, `milestone_reached` |
| `handle_account_state` | `renewal_due`, `winback_eligible`, `dormant_with_vera`, `gbp_unverified` |
| `handle_planning_intent` | `active_planning_intent` |
| `handle_recall` (customer) | `recall_due`, `appointment_tomorrow` |
| `handle_lapsed` (customer) | `customer_lapsed_soft`, `customer_lapsed_hard`, `trial_followup` |

### 3.7 `vera_bot/core/reply_handler.py`

State machine. Inputs: conversation history + new message. Output: `{action: send|wait|end, body?, cta?, rationale}`.

Detection order (first match wins):

1. **Opt-out / hostile** ‚Üí `end`
2. **Off-topic** (GST, tax, unrelated admin) ‚Üí `send` polite redirect once
3. **Auto-reply** (English markers OR Hindi markers OR same merchant turn hash 3+ times)
   - 1st detection ‚Üí one short owner-facing prompt
   - 2nd ‚Üí `wait 24h`
   - 3rd ‚Üí `end`
4. **Explicit commit** (yes/ok/go ahead/kar do/chalega) ‚Üí `send` action-mode draft confirm
5. **Ambiguous** ‚Üí re-enter `composer.compose` with a synthetic `reply_followup` trigger so the reply stays specific to the merchant's data instead of returning a generic line.

**Bug fix from v1:** v1 returned hardcoded "Want me to send the draft?" strings. v2 reuses the composer for the ambiguous branch ‚Äî keeps Specificity scores up in replay tests.

### 3.8 `vera_bot/core/language.py`

```python
def detect(text: str) -> Literal["en", "hi", "hi-en"]: ...
def prefers_hi_en(merchant: dict, customer: dict | None, recent_turn: str | None) -> bool: ...
```

`recent_turn` lets us flip language mid-conversation (brief ¬ß12.4). Detect Devanagari Unicode block + romanized Hindi token list (`hai`, `kar`, `aap`, `chahiye`, `kya`, `nahi`, `theek`, `accha`, `bahut`, `shukriya`).

### 3.9 `vera_bot/core/validators.py`

```python
def finalize(msg: ComposedMessage, ctx: ValidationContext) -> ComposedMessage:
    msg = strip_or_keep_urls(msg, policy=config.URL_POLICY)
    msg = trim_to_max(msg, config.MAX_BODY_CHARS)
    msg = enforce_single_cta(msg)
    msg = check_no_none_leak(msg)
    msg = anti_repeat_or_variant(msg, ctx.recent_bodies)
    return msg
```

**URL policy fix:** v1 stripped *all* URLs. Brief ¬ß5.4 allows URLs that add value. Default policy `digest_only`: keep at most one URL if it equals `category.digest[i].source_url` or `trigger.payload.source_url`. Never invent a URL.

### 3.10 `vera_bot/core/model_handlers.py` (optional)

Direct port of the Cloudera pattern, scaled down. **Off by default.** Used only by `validators.finalize` when:

- `LLM_ENABLED=true`, AND
- The deterministic handler returned the generic fallback branch, AND
- The body's "specificity score" (count of numbers/dates/source names) is below threshold.

```python
@dataclass
class ModelParameters:
    temperature: float = 0.0   # determinism contract
    top_p: float = 1.0
    max_tokens: int = 200

class UnifiedModelHandler:
    def __init__(self, inference_type: str, model: str, params: ModelParameters):
        self.inference_type = inference_type
        self.model = model
        self.params = params

    def generate(self, prompt: str) -> str:
        if self.inference_type == "openai":   return self._handle_openai(prompt)
        if self.inference_type == "gemini":   return self._handle_gemini(prompt)
        if self.inference_type == "bedrock":  return self._handle_bedrock(prompt)
        raise ValueError(f"unknown inference_type: {self.inference_type}")

def create_handler() -> UnifiedModelHandler:
    return UnifiedModelHandler(
        inference_type=config.LLM_PROVIDER,
        model=config.LLM_MODEL,
        params=ModelParameters(),
    )

def extract_json_from_text(text: str) -> dict:
    """Multi-strategy parse: json.loads ‚Üí ast.literal_eval ‚Üí regex. Cloudera-style."""
    ...
```

**What we deliberately drop from Cloudera:**
- Exponential backoff retry ‚Äî for the deterministic baseline, on LLM error we fall through to the deterministic body. No retry, no risk of blowing the 30s budget.
- ValidationException token-shrinking ‚Äî we're using `max_tokens=200`, not 100k+ contexts.
- Multi-provider concurrency probes ‚Äî single provider at a time.

### 3.11 `vera_bot/core/model_endpoints.py` (optional)

One function: `probe() -> dict[str, str]`. Sends a 10-token health prompt to the configured provider on app startup, logs `enabled` / `disabled`, and stores the result so `validators.finalize` knows whether to even try the LLM path. Never exposed as a route.

## 5. Judging rubric and how the architecture maps to it

| Rubric dimension | Where we win it |
|---|---|
| **Specificity** | `trigger_handlers` pulls numbers/dates/source names directly from context dicts; `validators.finalize` keeps source URLs under `digest_only` policy |
| **Category fit** | Per-category tone in `trigger_handlers` (clinical for dentists, operator-language for restaurants, ‚Ä¶) |
| **Merchant fit** | Composer reads `merchant.identity`, `performance`, `customer_aggregate`, `signals`; `language.py` honors per-merchant + per-turn language preference |
| **Trigger relevance** | One handler per trigger family; the trigger reason is named in the body, not implied |
| **Engagement compulsion** | `validators.enforce_single_cta` keeps one binary or one open-ended ask; handlers use loss-aversion / curiosity / effort-externalization framings |

## 6. Bugs from v1 that v2 fixes during the refactor

### 6.1 Suppression scope was global

`v1.bot.py:653` used `sent_suppression_keys: set[str]`. Brief shows category-shared keys like `"research:dentists:2026-W17"`. v2 stores `(merchant_id, suppression_key)` tuples in `core/store.py`.

### 6.2 URL policy contradicted the brief

`v1.bot.py:_strip_urls` removed all URLs. Brief ¬ß5.4 explicitly allows URLs when they add value. v2 default `URL_POLICY=digest_only`: a URL survives only if it appears in `category.digest[*].source_url` or `trigger.payload.source_url`. This recovers Specificity points (e.g. `JIDA Oct 2026 p.14` source link).

### 6.3 No anti-repetition

Brief ¬ß11 penalty. v2 adds `recent_bodies` deque per merchant in `core/store.py` and a variant request loop in `validators.finalize`.

### 6.4 Auto-reply detection was English-only

`v1.bot.py:_is_auto_reply` matched English markers. Brief Pattern B's example is Hindi (*"Aapki jaankari ke liye bahut-bahut shukriya"*). v2 `core/reply_handler.py` checks Devanagari + romanized Hindi markers, AND tracks merchant turn hashes for the "verbatim 3+" rule.

### 6.5 `/v1/reply` didn't reuse the composer

v1 returned hardcoded `"Want me to send the draft?"` lines. v2's reply handler routes the ambiguous branch back through `compose()` with a synthetic `reply_followup` trigger so replies stay specific to merchant data ‚Äî important for the top-10 replay test.

### 6.6 Language detection was once-per-merchant

v1 read `merchant.identity.languages` once. v2 `core/language.py` re-detects from each turn so a mid-flow Hindi switch is honored (brief ¬ß12.4).

### 6.7 Determinism was claimed but not enforced

`tools/validate_submission.py` runs `compose()` twice on every test pair and asserts byte-identical output. Catches accidental `datetime.now()` or unstable dict iteration before the judge does.

## 9. Testing checklist (expanded from v1 ¬ß12)

**Contract gates (must all pass before deploy):**
- Python syntax check on every module
- `expanded/` has 5 categories, 50 merchants, 200 customers, 100 triggers, 30 test pairs
- `submission.jsonl` has exactly 30 lines, each with `{test_id, body, cta, send_as, suppression_key, rationale}`
- Body length ‚â§ `MAX_BODY_CHARS`, no `None` substring leaks, no fabricated URLs
- Deterministic replay: every test pair runs twice, byte-identical
- All 5 endpoints return expected status + shape locally

**Behavior gates (worth building unit tests for):**
- Send same `(merchant_id, suppression_key)` twice ‚Üí second tick suppresses
- `research_digest` with empty `category.digest[]` ‚Üí no fabrication, returns "I'll wait" fallback
- Customer-scoped trigger without `CustomerContext` ‚Üí "missing customer" fallback fires (v1 already does this at `bot.py:516`; keep the regression test)
- Merchant with `languages=["en"]` only ‚Üí no Hindi tokens leak
- `chronic_refill_due` to non-pharmacy merchant ‚Üí routes to safe followup (v1 does this; regression test)
- Same body sent twice in a row ‚Üí anti-repetition kicks the second one to a variant
